# 生态议会运行时项目中期说明报告（评委版）

## 1. 项目背景与研究目标

本项目面向一个现实而又常见的问题：当社会舆论在讨论某一环境事件时，如何把“公众说法”与“物理世界中的可验证证据”放到同一个流程里核对，从而形成更可靠、更可追溯的研判结果。

围绕这一目标，我正在构建一个“监督器驱动的调查运行时”。它不是一个完全自由发挥的聊天式智能体，而是一套把任务分发、数据采集、证据整理、匹配核验、报告撰写和轮次推进串联起来的分阶段系统。项目希望解决三类问题：

- 让调查流程标准化，而不是依赖一次性人工操作。
- 让每一步都留下可回看、可复现、可审计的中间产物。
- 在保留大模型灵活性的同时，用规则、合同和文件边界约束它的行为。

## 2. 系统总体设计思路

系统将一次调查拆成若干清晰阶段，由“监督器”统一管理。

1. 主持人角色先明确本轮任务，只表达需要什么证据，不直接指定采集工具。
2. 社会学与环境学两个专家角色根据治理规则选择可用数据源。
3. 系统据此生成抓取计划，执行原始数据采集。
4. 进入确定性数据平面，把原始数据整理成公共主张、物理观测和匹配证据。
5. 专家撰写报告草稿，主持人形成阶段性决策。
6. 若证据仍不充分，系统进入下一轮并保留历史上下文。

这个设计的核心思想是把“可以自由判断的部分”和“必须稳定落盘的部分”分开：角色化专家负责受约束的判断，系统内核负责确定性加工、校验和留痕。

## 3. 工作流概览

对非技术读者而言，可以把该系统理解成一条分工明确的调查流水线。

### 第一层：任务设定

主持人先给出调查目标、地区、时间窗口和本轮任务清单。这里的重点不是直接下达“去抓哪个网站”，而是说明想验证什么、缺什么证据。

### 第二层：数据源选择

两个专家角色分别从公众信息侧和环境观测侧选择合适的数据源，但选择过程并不是任意的。系统会给出可用来源清单、治理边界、上轮记忆和当前任务需求，专家只能在被允许的范围内做选择。

### 第三层：数据采集

系统依据已经通过约束检查的选源结果，自动合成采集计划，并执行原始抓取。此阶段强调“原始数据先落盘”，避免后续步骤把采集过程和分析过程混在一起。

### 第四层：证据整理与核验

系统把公众侧信号整理成“候选主张”，把环境侧信号整理成“候选观测”，再用时间、空间和指标规则做匹配，生成支持、矛盾、混合或证据不足等结论。

### 第五层：报告与决策

专家根据结构化结果撰写报告，主持人决定本轮是否可以收束，还是需要下一轮继续补证。

## 4. 目前已完成的主要机制

截至当前阶段，项目已经实现了以下较成熟的基础能力：

- 监督器阶段机，能够明确区分本地阶段与代理阶段，并控制整轮推进。
- 分角色目录结构和规范文件路径，便于每类产物单独存储、调用和回放。
- 受治理约束的数据源选择机制，避免模型越权扩张数据范围。
- 从任务与选源结果确定性生成抓取计划的机制。
- 原始数据到候选主张、候选观测、匹配结果、证据卡片的确定性数据平面。
- 追加式审计链和快照留存机制，为关键阶段生成可追溯回执。
- 调查状态、后续行动建议、历史案例检索等辅助能力。
- 包结构、导入边界和调查基准的回归测试。

## 5. 当前客观评估

### 5.1 结构设计是否合理

总体上，当前结构设计是合理的，而且方向是正确的。

优点在于：

- 系统已经形成“任务设定、选源、抓取、规范化、匹配、报告、决策”的完整闭环。
- 根模块已被压缩为兼容入口，主要实现开始向 `application/`、`domain/`、`adapters/` 分层迁移。
- 关键文件路径被统一管理，便于定位某一步的输入与输出。

但也要客观看到，项目还处于“从第一阶段重构走向第二阶段稳定化”的过渡期：

- `controller/` 仍承担了较多枢纽职责，说明边界虽然比早期清楚，但还没有完全收敛。
- 部分应用层模块仍偏大，未来继续演化时存在再次形成“新单体模块”的风险。

因此，我的判断是：架构已经从“可运行原型”进入“有方法论约束的工程化原型”，但距离完全稳态架构还有一段工作要做。

### 5.2 可追溯与可审计性如何

这是当前项目最强的一部分之一。

系统已经具备较好的可追溯性基础：

- 每一轮、每一角色、每一阶段都有固定目录和标准产物。
- 关键阶段采用追加式审计链，回执中记录输入输出文件、哈希值和链式前驱关系。
- 抓取计划会记录输入快照，避免任务或选源变化后仍误用旧计划。
- 规范化、匹配和决策阶段都有明确的结构化中间文件，便于回放与复核。

不足之处在于：

- 审计强度在关键主流程上较强，但在“历史检索上下文”“调查动作排序依据”等辅助环节，还没有完全纳入同等强度的回执链条。
- 系统能证明“当时生成了什么”，但对“为什么偏向某个候选或某个行动”的解释力度还可以继续加强。

### 5.3 调查和推理能力如何

当前项目的调查能力是“流程强、边界清、启发式明确”，但还不是“高自主、高开放式探索”的形态。

它的优势在于：

- 能把舆论主张和物理观测放进同一调查流程中。
- 能围绕时间、空间和指标做基础证据核验。
- 能在证据不足时给出下一轮调查方向，而不是只产出一次性结论。

它的局限也很明显：

- 候选形成、匹配和后续动作推荐仍以规则和启发式为主。
- 大模型主要负责受约束的选择、筛选、审阅和撰写，而不是深度自主调查。
- 替代性假说虽然被显式保留，但目前对“反证”和“竞争性解释”的利用还不够强。

因此，我认为当前系统更像“可审计的调查流程骨架”，而不是已经成熟的“高级调查智能体”。

## 6. 主要风险与隐患

当前值得重点关注的风险包括：

- 规则覆盖风险：当环境事件类型继续扩展时，现有关键词、指标阈值和差距映射可能不够用。
- 启发式偏置风险：候选聚合、假说指派、行动排序带有人工设计痕迹，泛化能力有限。
- 模块再膨胀风险：若后续功能继续堆叠到少数枢纽文件，结构会再次恶化。
- 审计不均衡风险：主链条较强，辅助链条较弱，可能导致“结果可查，过程动机却不够透明”。
- 模型输出质量风险：虽然有 JSON 合同和校验器约束格式，但不能自动保证语义判断一定正确。

## 7. 下一阶段改进方向

下一阶段，我认为应重点沿四条线推进。

### 7.1 继续稳固结构边界

继续把过渡性的 `controller/` 职责外移到更稳定的应用层和适配层，降低中心枢纽压力。

### 7.2 补强审计闭环

把历史检索、调查动作生成、候选筛选理由等环节也纳入更统一的审计记录，提升“为何这样判断”的可复核性。

### 7.3 强化调查能力

这是下一阶段最值得投入的方向。重点包括：

- 让系统不仅会补证，还会主动区分“验证当前假说”和“尝试推翻当前假说”。
- 提高替代性假说的竞争强度，而不是只把它们作为附属备选项。
- 改进历史案例检索与跨轮记忆，让系统能更有效利用已归档案例支持新调查。

### 7.4 建立更严格的能力评测

围绕烟雾、洪水、高温等典型场景，持续建设基准样例和回归测试，使“调查能力是否真的变强”能够被系统化验证。

## 8. 中期结论

客观地说，这个项目当前最成熟的不是“智能”，而是“把智能活动放进一个可控、可追溯、可复盘的运行时框架里”。从研究型工程的角度看，这是一条合理且有价值的路线。

就中期状态而言，项目已经完成了比较扎实的系统骨架搭建，并在流程、边界、留痕和可复现性方面形成了明显特色。接下来的关键，不是继续无边界地增加功能，而是以现有骨架为基础，系统性增强调查能力、反证能力和跨轮学习能力，使该运行时从“规范化流程平台”进一步成长为“更强的环境事件调查系统”。
